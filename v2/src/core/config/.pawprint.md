# Config

A linked-list key-value store in SQLite. Each config entry is a row with a `next_id` pointer to its successor. Current value for a key: the row where `next_id IS NULL`. The index on `(key, next_id)` makes this lookup fast.

**Set**: Insert new row, update predecessor's `next_id` to point to it. Transactional.

**Undo**: Find current (head), find predecessor (whose `next_id` = current.id), clear predecessor's `next_id`, delete current. Constant-time, transactional. Multiple undos walk back through the chain.

**Source attribution**: Every entry records who made the change (cli, web, agent). The full history — what changed, when, by whom — is the data itself, not a separate audit log.

**Known keys**: `known_keys.ts` defines typed config entries with categories and validation. Unknown keys are allowed as `custom` with type inference. Known values get validated before storage (ranges, allowed values).

**Type system**: Values are serialized as strings in the DB, with a `type` column (string, integer, number, boolean) for deserialization. `parse_value.ts` / `serialize_value.ts` handle the round-trip.

The memory module's configurable parameters (k, min_score, half_life_days, candidate_pool_multiplier) resolve through this system via `resolve_config.ts`: call-site override > database config > hardcoded default.
