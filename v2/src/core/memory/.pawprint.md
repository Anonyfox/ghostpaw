# Memory

The memory system is the v2 module closest to production-complete. It implements the full belief lifecycle from MEMORY.md: storage with source-typed confidence, two-stage recall (SQL pre-ranking by `confidence * freshness`, then cosine similarity on the candidate set), evidence-based reinforcement, supersession chains for revised beliefs, and stale-memory surfacing for self-healing.

The search formula in `search_memories.ts` is the most important single function here: `confidence * exp(-(now - verified_at) / (86400000.0 * halfLife * sqrt(evidence_count)))`. The `sqrt(evidence_count)` factor means well-evidenced beliefs decay slower — a belief verified 9 times decays 3x slower than one verified once. This is the Ebbinghaus curve adapted for an agent that can actively reinforce memories.

`supersede_memories.ts` uses explicit BEGIN/COMMIT/ROLLBACK transactions and validates every ID before updating. It handles the "forgotten" sentinel case (self-referencing superseded_by when no replacement exists) cleanly.

All configurable parameters (k, min_score, half_life_days, candidate_pool_multiplier) resolve through `resolve_config.ts` with a clear precedence: call-site override > database config > hardcoded default. This means the ghost's recall behavior is tunable at runtime without code changes.

What's NOT here yet: the reconciliation mechanism (detecting contradictory beliefs and resolving them), the integration with the pack system (relational data exclusion per MEMORY.md's "What Memory Does Not Store" section), and the absorption pipeline (extracting beliefs from conversations — that's in v1's `absorb.ts` but hasn't been ported to v2's memory module yet).
