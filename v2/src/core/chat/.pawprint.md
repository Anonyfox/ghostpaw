# Chat

The mechanical turn execution pipeline. This module is a standalone machine — it takes a system prompt, tools, and model from above and operates the LLM loop. It does not assemble context, choose models, or make discretionary LLM calls. Those responsibilities belong to the harness layer.

The actual execution path from user input to stored response passes through ~200 lines across 6 files:

1. `execute_turn.ts` / `stream_turn.ts` — entry points. Acquire session lock, add user message, build chat, generate/stream, record result. Identical structure; execute returns a promise, stream yields chunks.
2. `build_chat.ts` — 25 lines. Creates LLM instance via chatoyant, sets system prompt, replays history, attaches tools. The system prompt enters the conversation at `chat.system(systemPrompt)` on line 13. This single line is where the ghost's identity reaches the model.
3. `record_turn.ts` — persists the assistant response, updates the session head pointer.
4. `add_message.ts` — inserts a message with parent linkage for the tree structure.
5. `get_history.ts` — retrieves the message chain via recursive CTE from the current head.
6. `acquire_session_lock.ts` — mutex per session ID, prevents concurrent turns on the same session.

The `systemPrompt` parameter is passed in from outside — nothing in this module assembles it. Context assembly (combining soul, memory, environment into the system prompt) lives in the harness layer. The string produced there ends up at `build_chat.ts:13`. That's the connection point.

Compaction trigger logic (`should_compact.ts`) lives here — it determines when history exceeds a token threshold. The actual LLM summarization call has moved to `harness/oneshots/summarize_for_compaction.ts`. The harness orchestrates: check trigger → run oneshot → then call execute/stream with compacted history.

Title generation has similarly moved to `harness/oneshots/generate_title.ts`. `derive_session_title.ts` (pure string truncation, no LLM) remains here.

The streaming path (`stream_turn.ts`) is an async generator that yields text chunks and returns the `TurnResult` on completion. Callers use `for await...of` and receive the final result from the generator's return value.

Both paths share error handling: LLM errors are caught and turned into a text response (`Error: <message>`) rather than thrown. The ghost tells you what went wrong instead of crashing.
